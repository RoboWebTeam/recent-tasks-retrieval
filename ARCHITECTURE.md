# Архитектура и бизнес-логика Roboweb

Этот файл — полное описание того, **как устроен проект и почему**, для разработчика, который впервые видит кодовую базу. Инструкции по запуску — в [GUIDE.md](./GUIDE.md), список технологий — в [TECH_STACK.md](./TECH_STACK.md).

---

## 1. Что делает продукт (бизнес-логика в двух словах)

Roboweb — сервис, который заменяет фрилансеров/конструкторы сайтов ИИ-чатом:

1. Пользователь регистрируется и попадает в **личный кабинет** (`/dashboard`).
2. Создаёт новый проект → открывается **редактор** (`/builder`) — это чат с ИИ слева и превью сайта справа.
3. Описывает идею словами («сделай сайт для кофейни с меню и доставкой») → backend отправляет запрос в Claude Sonnet 5 → получает готовый HTML-документ (всё в одном файле: `<style>` и `<script>` внутри) → показывает превью.
4. Пользователь просит правки («сделай кнопки синими», «добавь блок отзывов») — ИИ получает текущий HTML и вносит точечное изменение, не переписывая сайт с нуля.
5. Готовый сайт публикуется одним кликом на поддомене `roboweb.dev/site/<slug>`, либо на собственном домене клиента, либо экспортируется в GitHub Pages.
6. С сайта можно принимать заявки (форма), их видно в разделе «Заявки» (`/leads`) и в аналитике (`/analytics`).
7. Каждый AI-запрос списывает единицу из месячного лимита тарифа; при исчерпании лимита можно докупить «энергию» (пакеты доп. запросов) через ЮKassa.

## 2. Общая архитектура (5 слоёв)

```
┌─────────────────────────────────────────────────────────┐
│  1. Frontend (src/)                                       │
│     React SPA (Vite). Общается с backend только по HTTP.  │
└───────────────────────┬─────────────────────────────────┘
                         │  fetch() на URL из src/lib/apiConfig.ts
┌───────────────────────▼─────────────────────────────────┐
│  2. Backend (backend/*)                                    │
│     Python 3.11 cloud-функции. Каждая папка = 1 функция.   │
│     Доступ к DB (DATABASE_URL) и Secrets (os.environ)      │
└──────┬─────────────────────┬─────────────────┬───────────┘
       │                     │                 │
┌──────▼──────┐   ┌──────────▼──────────┐  ┌───▼────────────┐
│ 3. PostgreSQL│   │ 4. Секреты (env)     │  │ 5. S3 (reg.ru)  │
│ (db_migrations)│  │ ключи API, пароли   │  │ файлы/картинки │
└─────────────┘   └──────────────────────┘  └────────────────┘
```

**Жёсткое правило:** frontend НИКОГДА не обращается к базе данных или S3 напрямую — только через backend-функции по HTTP. Секреты никогда не попадают во frontend-код.

## 3. Frontend: карта страниц и что они делают

Роутинг определён в `src/App.tsx`. Все страницы, кроме `Index`, подгружаются лениво (`lazy()`), чтобы не раздувать первый бандл.

### Публичные страницы (без авторизации)

| Маршрут | Файл | Назначение |
|---|---|---|
| `/` | `pages/Index.tsx` | Главная маркетинговая страница: hero, демо-режим, портфолио примеров, тарифы, FAQ |
| `/login`, `/register` | `pages/Login.tsx`, `Register.tsx` | Вход и регистрация (email/пароль, GitHub OAuth, Яндекс OAuth) |
| `/blog`, `/blog/:slug` | `pages/Blog.tsx`, `Article.tsx` | Блог компании (SEO-контент, статьи о продукте) |
| `/pricing`, `/pricing/status` | `pages/Pricing.tsx`, `OrderStatus.tsx` | Тарифы и статус оплаты после ЮKassa |
| `/site/:slug` | `pages/PublicSite.tsx` | **Публичная витрина** — здесь открывается опубликованный сайт конкретного пользователя (просто рендерит HTML из БД) |
| `/oferta`, `/privacy`, `/personal-data` | `pages/legal/*` | Юридические документы |
| `/auth/github/callback`, `/auth/yandex/callback` | `GithubCallback.tsx`, `YandexCallback.tsx` | OAuth redirect-обработчики |

### Защищённые страницы (требуют сессию, обёрнуты в `<ProtectedRoute>`)

| Маршрут | Файл | Назначение |
|---|---|---|
| `/dashboard` | `pages/Dashboard.tsx` | Личный кабинет: три вкладки — **Проекты** (`DashboardProjectsTab`), **Тариф** (`DashboardPlanTab`), **Профиль** (`DashboardProfileTab`) |
| `/builder` | `pages/Builder.tsx` | **Главный экран продукта** — редактор сайтов (см. раздел 4) |
| `/settings/domain` | `pages/DomainSettings.tsx` | Подключение и верификация собственных доменов |
| `/analytics` | `pages/Analytics.tsx` | Просмотры страниц, устройства, источники трафика по сайтам пользователя |
| `/leads` | `pages/Leads.tsx` | Заявки, собранные формами на опубликованных сайтах |
| `/files` | `pages/Files.tsx` | Файловое хранилище пользователя (картинки, HTML, ZIP) |

### Служебная страница

| Маршрут | Файл | Назначение |
|---|---|---|
| `/admin` | `pages/Admin.tsx` | Админ-панель, защищена собственным паролем (`ADMIN_KEY`) внутри компонента, а не через сессию пользователя. Вкладки: аналитика по всем сайтам, лиды, email-подписчики, пользователи, лог действий, уведомления, тарифы, чат поддержки — компоненты в `pages/admin/*` |

## 4. Редактор сайтов (`Builder.tsx`) — сердце продукта

Это самый сложный компонент проекта. Layout: слева чат с ИИ (можно свернуть/растянуть мышкой), справа — превью/код/ядро проекта (переключатель вкладок).

### Поток генерации сайта

1. Пользователь пишет сообщение в чат → `sendMessage()` в `Builder.tsx`.
2. Frontend отправляет POST на функцию `generate-site` с полями: `messages` (история чата), `project_id`, `current_html` (если сайт уже существует — это признак **правки**, а не создания с нуля), `style` (выбранный пресет дизайна, если новый сайт).
3. Backend (`backend/generate-site/index.py`):
   - Проверяет и списывает квоту AI-запросов пользователя.
   - Определяет, «крупная» ли задача (`detect_large_task` — по ключевым словам типа «магазин», «каталог», «лендинг») — крупная задача стоит 3 единицы квоты вместо 1 и получает больше токенов на генерацию.
   - Собирает системный промпт (`SYSTEM_PROMPT` — детальные требования к дизайну, тексту, SEO, структуре HTML) + при правке — отдельно текущий HTML и точную инструкцию «внеси только это изменение, остальное сохрани».
   - Отправляет запрос напрямую в **Anthropic API** (`claude-sonnet-5`), без посредников.
   - Проверяет, что ответ — целостный HTML (заканчивается на `</html>`, есть видимый контент в `<body>`), при обрыве — либо чинит теги, либо просит повторить (квота возвращается, если результат невалиден).
   - При создании нового сайта (не правки) и достаточном запасе времени — запускает **второй проход**: просит модель критически улучшить только что сгенерированный сайт (арт-директорский самоконтроль).
   - Сохраняет готовый HTML в колонку `projects.html_content`.
4. Frontend получает HTML, обновляет `<iframe>` превью, добавляет ответ ассистента в чат с кратким отчётом о сделанном.

### Важные технические детали редактора

- **HTML хранится целиком в PostgreSQL** (`projects.html_content`, тип `TEXT`), а не как файл в S3. Так исторически сложилось и осознанно не менялось — это простое и надёжное решение, миграция на файловое хранение рискованна для уже опубликованных сайтов.
- **Модель ИИ** — только Claude Sonnet 5 через прямой Anthropic API. Раньше использовался OpenRouter как прокси для нескольких моделей (Gemini/GPT-4o/Claude/Opus) — эта возможность выбора модели была убрана, весь код теперь заточен под единственную модель.
- **Таймаут функции** — Sonnet 5 генерирует небыстро, функция `generate-site` требует повышенного таймаута (минимум 60, оптимально 90 секунд); переменная `FUNCTION_TIMEOUT_SEC` должна совпадать с реальной настройкой таймаута функции, иначе внутренние расчёты токенов/времени будут неверными.
- **Вкладки правой панели**: `preview` (живой iframe с сайтом), `code` (просмотр/ручное редактирование HTML), `core` (см. ниже).
- **Ядро проекта** (вкладка `core`, компонент `BuilderCorePanel.tsx`) — доступ к секретам, собственной пользовательской БД проекта (`project_db_tables`/`project_db_rows`), файловому хранилищу и функциям проекта. Это песочница для более продвинутых пользователей.
- **Визуальный редактор** — режим клика по элементам прямо в iframe для точечного изменения текста/стилей без диалога с ИИ.
- **История версий** — последние версии HTML хранятся в состоянии компонента (не в БД) для отката без повторного обращения к ИИ.
- **Публикация** — кнопка «Опубликовать» вызывает `apiPublishProject`, которая генерирует уникальный `slug` и меняет `status` проекта на `published`; после этого сайт доступен по `/site/<slug>` и отдаётся функцией `public-site` (публичной, без авторизации).

## 5. Backend: полный список функций

Backend — НЕ монолит. Каждая папка в `/backend/*` — независимая serverless-функция на Python 3.11 с точкой входа `def handler(event, context) -> dict`. Функции не могут напрямую вызывать друг друга — только через HTTP, если это нужно.

| Функция | Назначение |
|---|---|
| `auth` | Регистрация, вход, OAuth (GitHub/Яндекс), смена пароля, удаление аккаунта, управление сессиями |
| `generate-site` | Генерация и правка HTML сайтов через Claude Sonnet 5 (см. раздел 4) |
| `user-projects` | CRUD проектов пользователя (создание, список, получение по id, удаление, история чата) |
| `project-core` | «Ядро» проекта: секреты, пользовательские таблицы БД, файлы, функции проекта |
| `site-files` | Загрузка/удаление файлов пользователя в S3 (reg.ru), список файлов |
| `public-site` | Отдаёт HTML и SEO-метаданные опубликованного сайта по `slug` — без авторизации |
| `domains` / `verify-domain` | Подключение собственных доменов, проверка DNS (A/CNAME через Google DNS-over-HTTPS), SSL-статус |
| `github-publish` | Экспорт HTML сайта в репозиторий GitHub Pages |
| `site-leads` / `get-leads` | Заявки, оставленные посетителями на сайтах клиентов / email-подписчики главной страницы |
| `plan-pricing` / `energy-pricing` | Цены тарифов Pro и пакетов «энергии» (докупка AI-запросов); чтение публично, изменение — только с `X-Admin-Key` |
| `order-status` | Проверка статуса конкретного платежа по номеру заказа |
| `send-email` | Отправка транзакционных писем через SMTP Яндекс.Почты |
| `support-chat` | Виджет чата техподдержки (диалоги, сообщения с файлами, быстрые ответы, рабочие часы) |
| `analytics` | Запись просмотров страниц, определение устройства по User-Agent, агрегированная статистика |
| `activity-log` | Журнал действий (регистрации, генерации, смены тарифа, блокировки) для админ-панели |
| `manage-user` | Административные операции над пользователями: блокировка, смена плана, удаление |
| `blog-rss` | Генерация RSS-ленты блога (используется для публикации в Яндекс.Дзен) |
| `extensions/yookassa/yookassa` | Создание платежа в ЮKassa, возврат `payment_url` |
| `extensions/yookassa/yookassa-webhook` | Обработка webhook об оплате: подтверждение, активация тарифа/энергии, отправка чека |

Соответствие имя-функции → публичный URL хранится в `backend/func2url.json` (автогенерируется) и используется на фронте через `src/lib/apiConfig.ts`.

## 6. База данных — ключевые таблицы

Схема управляется миграциями `db_migrations/V####__описание.sql`, применяются строго по возрастанию номера.

| Таблица | Назначение |
|---|---|
| `users` | Аккаунт: email, пароль (хеш), тариф (`plan`), лимит/использовано AI-запросов, энергия, GitHub-токен, блокировка |
| `sessions` | Токены сессий (криптостойкий id, срок жизни 30 дней) |
| `projects` | Сайты пользователя: `title`, `status` (draft/building/published), `slug`, **`html_content`** (весь HTML), `chat_history` (JSONB), поля для GitHub-публикации |
| `site_files` | Метаданные файлов пользователя (сам файл — в S3, тут только `file_key`/`file_url`) |
| `domains` | Подключённые собственные домены, статус DNS/SSL |
| `orders` / `order_items` | Платежи через ЮKassa (покупка тарифа или энергии) |
| `page_views` | Просмотры страниц опубликованных сайтов (для аналитики) |
| `site_leads` | Заявки с форм на сайтах клиентов |
| `leads` | Email-подписка на главной странице (не путать с `site_leads`) |
| `activity_log` / `admin_notifications` | Журнал действий и уведомления для админки |
| `plan_pricing` / `energy_pricing` / `energy_packages` | Цены тарифов и пакетов энергии |
| `support_conversations` / `support_messages` / `support_quick_replies` / `support_settings` | Чат техподдержки |
| `project_secrets` / `project_db_tables` / `project_db_rows` | «Ядро» проекта — пользовательские секреты и мини-БД внутри конструктора |

**Важно про хранение HTML:** сайты хранятся как текст в `projects.html_content`, а не файлами в S3. Это осознанное решение — миграция на файловое хранение затронула бы все уже опубликованные сайты и была признана слишком рискованной без явной необходимости.

## 7. Монетизация: тарифы, энергия, квоты

- **Тарифы** (`users.plan`): `free`, `premium`, `pro_60`, `pro_80`, `pro_200`, `pro_400`, `pro_800` — число в имени Pro-тарифов означает месячный лимит AI-запросов.
- Каждый успешный вызов `generate-site` списывает `requests_used` (1 единица за обычный запрос, 3 — за «крупную задачу», см. `detect_large_task`).
- При исчерпании месячного лимита можно докупить **энергию** (`energy_balance`) — разовые пакеты доп. запросов, которые не сгорают в конце месяца.
- Лимит сбрасывается ежемесячно (`requests_reset_at`).
- Порог `LOW_BALANCE_THRESHOLD = 5` — при остатке ≤5 запросов пользователю показывается предупреждение и по email (`maybe_notify_low_balance`, один раз через флаг `low_balance_notified`).
- Оплата — ЮKassa: создание платежа (`extensions/yookassa/yookassa`) → редирект пользователя → webhook (`yookassa-webhook`) подтверждает оплату, активирует тариф/энергию, отправляет чек по 54-ФЗ.

## 8. Файловое хранилище (S3)

Два независимых хранилища сосуществуют:

1. **Встроенное S3 платформы** (`AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY`, endpoint `bucket.poehali.dev`) — легаси, использовалось раньше для всех файлов. Сейчас на нём остаются только старые статические картинки (обложки блога, примеры портфолио на главной) — их перенос не выполнялся осознанно (см. п. 9).
2. **Собственное хранилище на reg.ru** (Рег.облако, `REG_S3_ACCESS_KEY_ID`/`REG_S3_SECRET_ACCESS_KEY`, endpoint `https://s3.regru.cloud`, bucket `roboweb`) — используется для **всех новых загрузок**: файлы пользователей (`site-files`) и вложения в чате поддержки (`support-chat`).

**Критичный нюанс reg.ru:** объект нужно загружать с явным `ACL='public-read'` в `put_object` — иначе даже у «публичного» bucket прямая ссылка на файл вернёт 403 Forbidden.

## 9. Известные технические ограничения и решения

Эти решения были приняты осознанно в ходе развития проекта — если планируете их менять, учитывайте контекст:

- **HTML сайтов остаётся в БД, а не в S3** — перенос признан слишком рискованным для живых опубликованных сайтов без явной необходимости.
- **Старые картинки блога/портфолио остаются на встроенном S3 платформы** — их физический перенос на reg.ru невозможен в среде разработки без инструмента скачивания бинарных файлов; решено не трогать, пока они работают.
- **Единственная модель ИИ — Claude Sonnet 5 напрямую** — раньше через OpenRouter было можно выбирать между Gemini/GPT-4o/Claude/Opus, но при переходе на прямой Anthropic API возможность выбора модели была сознательно убрана (один провайдер — меньше точек отказа и затрат на прокси).
- **Функции `generate-image` и `ai-balance` удалены** — генерация изображений и мониторинг баланса OpenRouter больше не нужны после отказа от OpenRouter.

## 10. Интернационализация и SEO

- `src/lib/i18n.ts` — переключение RU/EN на уровне интерфейса (не касается сгенерированных сайтов пользователей — их язык задаётся отдельно в промпте генерации).
- `src/lib/seo.ts` — программная установка `<title>`, `<meta description>`, Open Graph, JSON-LD (Article/Blog schema) для страниц блога и главной — важно для органического трафика.
- `backend/blog-rss` — генерирует RSS специально под требования Яндекс.Дзен.
